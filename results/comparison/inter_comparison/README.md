# Inter-codebook comparison figures

Four figures that put every 3GPP PMI codebook family **on the same axes**, scored
the way the codebooks were actually justified during standardization — not by a
single similarity number, but by *feedback overhead versus multi-user throughput*,
with realism knobs (CSI aging, beamformed CSI-RS, receiver model, rank adaptation)
that each family was designed around.

Generated by [`scripts/compare/inter_comparison.py`](../../../scripts/compare/inter_comparison.py):

```bash
.venv/bin/python scripts/compare/inter_comparison.py            # this folder's figures
.venv/bin/python scripts/compare/inter_comparison.py --quick    # CI smoke variant
```

Every number quoted below is in [`summary.json`](summary.json), written by the same run.

---

## 1. Why compare this way

**The problem with the obvious metrics.** The two quantities most papers report —
SGCS and single-user spectral efficiency — are both poor *discriminators*:

* **SGCS** (squared generalized cosine similarity) measures how well the reported
  precoder aligns with the channel's ideal eigen-directions. It is cheap and it is
  the agreed *intermediate* KPI of the 3GPP AI/ML CSI study (TR 38.843), but it says
  nothing about what happens when the precoder is *used* against other users'
  interference, and it weights all subbands/layers equally regardless of the rate
  they carry.
* **Single-user (SU) SE** with a genie receiver is extremely forgiving: at 10 dB,
  a 23-bit Type I report already reaches 93% of channel capacity and a 225-bit
  Type II reaches 98.5% (Figure 3, right panel). If you compared codebooks only on
  SU SE you would conclude Type II is not worth 10× the bits — which contradicts
  why it was standardized.

**What actually separates codebooks is MU-MIMO.** When the gNB co-schedules K
users on the same resources, it uses the reported precoders to *zero-force*
(spatially null) each user's signal against the others. Any error in the reported
direction becomes **residual inter-user interference** that does not vanish with
SNR. This is the regime Type II (and everything after it) exists for, and it is the
axis RAN1 used to select these codebooks: system-level user throughput (mean and
cell edge) **versus feedback overhead**, compared at equal overhead or as a curve.
Figure 1 is exactly that comparison; Figures 2–4 add the realism axes (aging,
beamformed ports, receiver model, rank choice) that the sweep-style single-number
evaluations miss.

**Matched drops.** Within each figure, every scheme is evaluated on *identical*
channel realizations (same RNG seed). Differences between curves are therefore
differences between codebooks, not between channel draws. Mean-only, unmatched
comparisons with 6 drops are how codebook myths get made; here each MU point
carries a Student-t 95% confidence interval over drops.

## 2. Common setup

| Parameter | Value | Meaning |
|---|---|---|
| Antenna array | N1×N2 = 4×2, dual-polarized → **P = 16 CSI-RS ports** | gNB transmit array; every codebook family supports this geometry, so all are comparable |
| PMI frequency units | **N3 = 8** | number of subbands the PMI report covers |
| Channel | `RandomRayChannel`, **4 paths**, off-grid angles/delays | sparse multipath, frequency-selective; drawn fresh per drop |
| UE antennas | n_rx = 2 (1 for the MU figure: one stream per user) | limits usable rank |
| Working SNR | **10 dB** | mid-range operating point; SE and MU rates quoted here |
| Monte-Carlo drops | **40** per (scheme, point), matched seed 0 | one drop = one channel realization |
| Evaluation | `nr_csi.eval.evaluate` / `evaluate_mu` | genie CSI at the UE (no estimation noise); selection and scoring on the same realization except where delay is swept |

**The codebook families** (fixed color per family in *all* figures):

| Family | Release | One-line description | Feedback range here |
|---|---|---|---|
| **Type I** (blue) | R15 | one DFT beam per polarization + wideband co-phase; the baseline "direction only" report | ~23 bits |
| **Type II** (green) | R15 | L ∈ {2,3,4} DFT beams linearly combined with per-subband amplitude/phase coefficients | 116–226 bits |
| **eType II** (yellow) | R16 | Type II with the per-subband coefficients compressed on an FD (DFT) basis; `pc2/pc4/pc6` are spec parameter combinations (L, p, β) controlling beams/FD-taps/coefficient budget | 47–159 bits |
| **FeType II PS** (red) | R17 | *port-selection*: the UE picks among (beamformed) CSI-RS ports instead of DFT beams and reports a small combining matrix; `pc2/pc4` set (M, α, β) | 34–46 bits (beamformed) |
| **R18 Doppler** (violet) | R18 | eType II extended with a Doppler-domain basis: one report carries **predicted precoders for N4 future slot intervals** | ~248 bits |
| **full CSI** (gray dashed) | — | unquantized eigen-direction feedback, same ZF pipeline: the feedback-free ceiling | ∞ |

**Feedback bits** = the UCI payload of one PMI report, counted per the spec's own
field tables (e.g. beam-combination index, coefficient bitmaps, amplitude/phase
indicators). For Type II-class codebooks the count varies slightly with the
realization because only nonzero coefficients are reported — the value plotted is
the mean over drops.

## 3. Figure 1 — `fig1_mu_pareto.png`: MU-MIMO overhead/throughput Pareto

**What is "MU ZF sum rate"?** K = 4 users each experience an independent channel
drop and report a rank-1 PMI on it. The gNB stacks the 4 reported directions and
**zero-forces (ZF)**: it computes the pseudo-inverse of the stacked directions, so
each user's beam has (ideally) zero projection onto the other users' *reported*
directions, and splits transmit power equally across the 4 streams. Each user's
rate is then computed against their **true** channel — including the residual
interference that leaks because the reported direction ≠ true direction. The **sum
rate** is the total over the 4 users, averaged over 40 matched drops
(`nr_csi.eval.evaluate_mu`). Quantization error → imperfect nulls → interference
that persists at any SNR: this is the mechanism that makes feedback resolution
valuable, and it is exactly what SU evaluation cannot see.

**What is a Pareto front?** Each codebook configuration is a point (feedback bits,
sum rate). A point is *Pareto-dominated* if another point has both fewer bits and a
higher rate. The interesting comparison is not "which codebook is best" (more bits
almost always help) but **which family's curve lies above the others at equal
overhead** — the overhead-efficiency frontier.

**Panels.** Left: mean sum rate ±95% CI vs bits, with the full-CSI ZF ceiling.
Right: **cell-edge proxy** — the 5th percentile of the *pooled* per-user rates
(all drops × all users into one CDF, then p5), mirroring how system-level
evaluations report edge-user throughput.

**Results and reading:**

| point | bits | sum rate | p5 user rate |
|---|---|---|---|
| Type I | 23 | 10.70 ± 0.74 | 1.13 |
| eType II pc2 / pc4 / pc6 | 47 / 86 / 159 | 13.01 / 14.68 / **16.11** | 1.57 / 2.27 / 2.55 |
| Type II L=2 / L=3 / L=4 | 116 / 156 / 226 | 13.79 / 14.80 / 15.87 | 1.99 / 2.30 / 2.60 |
| FeType II (beamformed) pc2 / pc4 | 35 / 46 | 13.01 / 13.11 | 1.54 / 1.59 |
| full-CSI eigen ZF | — | 18.91 | — |

* **eType II Pareto-dominates Type II**: pc4 beats L=2 with 30 fewer bits; pc6
  beats L=4 with 67 fewer bits. This is the Rel-16 standardization result ("same
  throughput at roughly half the overhead") reproduced from scratch.
* **Type I → Type II is worth 3–5 bits/s/Hz here** while being nearly invisible in
  SU terms (Figure 3): the entire *raison d'être* of high-resolution feedback in
  one figure pair.
* **The cell edge amplifies the differences** (p5: 1.13 → 2.6, a 2.3×): imperfect
  nulls hurt the weakest co-scheduled user the most.
* The ~2.8 bits/s/Hz gap from the best quantized point to the full-CSI ceiling is
  the residual quantization-interference floor at K = 4.

## 4. Figure 2 — `fig2_delay.png`: CSI aging and the R18 Doppler codebook

**What is CSI aging?** A PMI is measured at slot t but applied at slot t + d
(feedback decoding, scheduling latency). On a time-varying channel the report is
stale by the time it is used. The x-axis sweeps this **feedback delay d** (in slot
intervals); `nr_csi.eval.delay_sweep` re-seeds identically per delay, so all
delays see the *same* channel processes and the curve isolates pure aging.

**Setup.** Fast fading (`max_doppler = 1.0`, period 8 slots — a large fraction of
a Doppler cycle inside the prediction window), rank 1, 40 drops. Static codebooks
observe a 4-slot measurement window (same observation length as R18, so noise
averaging is not a confound); the **R18 Doppler** codebook reports predicted
precoders for its N4 = 4 future intervals and the gNB applies the one matching
each slot's actual age (`delay_aware=True`).

**Results and reading** (SGCS at d=0 → d=6):

* R18 Doppler (248 b): **0.933 → 0.728** — flattest decay by far.
* eType II (156 b): 0.893 → 0.647; Type II (225 b): 0.878 → 0.646 — high-resolution
  static reports age fast.
* Type I (23 b): 0.671 → 0.658 — a coarse wideband beam has almost nothing to lose.

Three textbook effects, none programmed in:

1. **Prediction pays inside its horizon**: R18's advantage is widest at d = 1…4
   (its N4 window) and narrows beyond it, where the codebook can only clamp to its
   last predicted interval.
2. **The SE curves cross**: beyond d ≈ 4, 23-bit Type I outperforms 225-bit
   Type II. Expensive feedback is worthless when stale — a classic system-design
   result.
3. At d = 6 all static codebooks converge to SGCS ≈ 0.65 regardless of overhead.

**Fair-bits caveat:** R18 carries 248 bits vs eType II's 156, so part of its d=0
head start is overhead, not prediction. The honest signal is the **slope**
(R18 loses 0.21 SGCS over 6 slots; eType II loses 0.25 over just 2), not the d=0
offset.

## 5. Figure 3 — `fig3_rate_distortion.png`: SU rate-distortion and the port-selection physics

**What is a rate-distortion view?** Feedback design is lossy compression of the
channel's dominant eigenvectors: x-axis = bits spent, y-axis = fidelity achieved.
Left panel uses **SGCS** as fidelity (1.0 = perfect alignment); right panel uses
**SE / capacity** — achieved single-user rate at 10 dB divided by the waterfilling
capacity of the same channel at the same rank (the provable ceiling for *any*
precoder), so 1.0 means "feedback cost nothing".

**Why FeType II appears twice.** Port-selection codebooks are specified for
**beamformed CSI-RS**: the gNB precodes each CSI-RS port with a spatial beam
(derived from uplink SRS via reciprocity), and the UE merely *selects ports* and
reports a small combining matrix — that's why their payloads are small. Evaluating
them on raw antenna elements (dashed) makes "port selection" mean "pick bare array
elements", which measures the missing beamforming, not the codebook. The solid red
curve uses `nr_csi.eval.BeamformedPortsScheme`: the gNB applies the top-4
per-polarization **eigenbeams** of the wideband channel covariance to the CSI-RS
(orthonormal, so transmit power conventions are preserved), the UE runs the
unmodified R17 codebook on the 8 effective ports, and the resulting precoder is
mapped back to the 16 physical ports for scoring — so both variants are judged
against the same raw channel.

**Results and reading:**

* Same codebook, same channel: raw pc2 = 73 bits → SGCS **0.526**; beamformed
  pc2 = 34 bits → SGCS **0.785**. Beamformed pc2 beats raw pc4 (113 b → 0.770)
  with less than a third of the bits. One picture of why the earlier exhaustive
  sweeps made every port-selection codebook look broken: the evaluation physics
  was wrong, not the implementations.
* **eType II dominates Type II per bit** across the whole curve (pc4: 0.880 at
  86 b vs L=3: 0.882 at 155 b) — consistent with Figure 1 and with the 70k-run
  exhaustive sweeps.
* The right panel shows the **compression of SU SE**: every family ≥ 0.93 of
  capacity. Read side-by-side with Figure 1, this is the argument for MU-based
  comparison in one glance.

**Caveat:** beamformed FeType II saturates near SGCS ≈ 0.79, below eType II pc4.
Partly real (4 eigenbeams per pol cap the achievable subspace on a 4-path
frequency-selective channel; pc2/pc4 have only M = 1 FD tap), partly an adapter
limitation: the R17 design also assumes gNB *delay pre-compensation* per port,
which `BeamformedPortsScheme` does not model. Treat the solid red curve as a lower
bound on R17's intended performance; the raw-vs-beamformed *contrast* is the
reliable finding.

## 6. Figure 4 — `fig4_receiver.png`: receiver model and rank adaptation

**Left panel — what "joint decoding" vs "per-layer MMSE" means.** At rank 2 the
precoder carries two spatial layers. The standard SE formula
(log₂ det(I + ρ·(HW)(HW)ᴴ), solid bars) assumes the receiver decodes the layers
*jointly* — inter-layer interference costs nothing. Real link adaptation decodes
each layer through a **linear MMSE filter that treats the other layer as noise**
(hatched bars, `nr_csi.metrics.su_rate_mmse`): the rate is Σ log₂(1 + SINR_l) with
SINR_l = 1/[(I + ρ·GᴴG)⁻¹]_ll − 1, G = HW. The gap between the bars is the price
of **non-orthogonal effective layers** — invisible to SGCS and to the joint-decoding
SE, but paid by real receivers.

**Results** (rank 2, 10 dB, 6-path channel):

| family | joint | per-layer MMSE | loss |
|---|---|---|---|
| Type II | 10.48 | 10.32 | 0.16 |
| eType II | 10.46 | 10.25 | 0.21 |
| Type I | 8.69 | 8.02 | **0.67** |
| FeType II (beamformed) | 9.68 | 8.76 | **0.93** |

Type II-class codebooks quantize both eigenvectors well, so their effective layers
stay near-orthogonal and the receiver model barely matters. Type I's **rigid,
predefined rank-2 beam pair** cannot rotate to the channel's eigenbasis — the same
rigidity that shows up as a rank-2 SGCS collapse in the exhaustive sweeps — and
loses 4× more. FeType II's per-port combining constrains layer orthogonality most.

**Right panel — auto-RI.** Real UEs report a rank indicator (RI) along with the
PMI, choosing the rank that maximizes their expected rate; comparing codebooks at
a fixed rank distorts families whose rank-2 structures differ in flexibility.
`evaluate(..., rank="auto")` makes the rank decision per drop from the measured
channel at the working SNR. On this rich channel at 10 dB, auto-RI picks rank 2 in
95–100% of drops and its SE sits exactly on max(rank 1, rank 2) for every family —
the mechanism validated (auto can never do worse than the best fixed rank on
matched drops; see `tests/eval/test_auto_rank.py`). At lower SNR or on sparser
channels the auto marker separates from both fixed markers.

## 7. Glossary

* **PMI / RI / UCI** — Precoder Matrix Indicator / Rank Indicator, the codebook
  index the UE feeds back; UCI is the uplink control payload carrying them.
* **SGCS** — squared generalized cosine similarity between reported and ideal
  precoder columns, |w_refᴴ·w|² / (‖w_ref‖²‖w‖²), averaged over layers and
  subbands; ∈ [0, 1]; the 3GPP AI/ML-study intermediate KPI.
* **SE (spectral efficiency)** — achievable rate in bits/s/Hz from the log-det
  formula, noise power 1, total transmit power ρ (tr(WᴴW) = 1 convention).
* **Capacity bound** — waterfilling over the channel's top-r singular values; the
  supremum over all rank-≤ r precoders. "SE / capacity" = feedback efficiency.
* **ZF (zero-forcing)** — multi-user precoding that inverts the stacked reported
  directions so each stream nulls the others' *reported* channels.
* **Sum rate** — total rate of the K co-scheduled users, interference included.
* **Cell edge / p5** — 5th percentile of the pooled per-user rate distribution;
  the "unlucky user" statistic system evaluations care about.
* **Pareto front** — the set of (bits, rate) points not beaten on both axes
  simultaneously; the overhead-efficiency frontier.
* **95% CI** — Student-t confidence interval on the mean over the 40 matched
  drops (`nr_csi.eval.mean_ci`).
* **Drop** — one independent channel realization in the Monte-Carlo loop.
* **CSI aging / feedback delay** — slots between measuring the channel and
  applying the report.
* **Beamformed CSI-RS** — CSI-RS ports pre-precoded with gNB-side spatial beams;
  the operating assumption of every port-selection codebook.
* **Eigenbeam** — dominant eigenvector of the wideband spatial covariance; the
  optimal wideband beam.
* **FD basis / M taps** — the DFT-in-frequency compression of per-subband
  coefficients introduced in R16 (and used with M ∈ {1, 2} in R17).
* **N4 / DD basis** — the R18 time-domain analogue: one report spans N4 slot
  intervals via a Doppler-domain basis, i.e. channel prediction.
* **auto-RI** — per-drop rank selection maximizing measured-channel rate at the
  working SNR (`rank="auto"`).

## 8. Reproducibility and caveats

* Deterministic: seed 0, matched across schemes per figure; rerunning the command
  reproduces every number in `summary.json` exactly. `--figs 1,3` rebuilds a subset;
  `--n1/--n2/--n3/--users/--drops/--seed` change the operating point.
* Backing tests (all invariant- or closed-form-anchored):
  `tests/eval/test_mmse_rate.py`, `test_mu_stats.py`, `test_beamformed.py`,
  `test_auto_rank.py`, `test_delay_sweep.py`, `test_inter_comparison_script.py`.
* Known limitations: single-cell, no scheduling/link-adaptation loop (this is a
  link-level proxy for SLS, not SLS); genie channel knowledge at the UE
  (no estimation noise — the harness supports it via `measurement_snr_db`, unused
  here); FeType II adapter models spatial beamforming only (no per-port delay
  pre-compensation), so R17's absolute position is a lower bound; R18 vs static
  comparison in Figure 2 is not bit-matched (read slopes, not offsets); synthetic
  ray channel rather than 3GPP CDL (the repo's Sionna CDL channel can be swapped
  in, at ~10× runtime and without seed-reproducibility).
